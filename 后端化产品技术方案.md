# Pantry 后端化产品技术方案（v1/v2）

> 目的：把当前“纯前端 + localStorage”的项目，升级为“有后端、开箱可用、可跨端共享”的产品方案。  
> 本文是后续 coding agent 的实施依据（先方案、后编码）。

---

## 1. 已确认的产品决策（来自当前需求）

1. **完全不登录**（无邮箱/手机号/OAuth）。
2. **版本 1（先上线）**：以浏览器为单位，每个浏览器默认绑定一个“家庭（family）”。
3. **版本 2（共享）**：支持多个设备加入同一个家庭，且**全部可编辑**（不做只读权限）。
4. **AI 识别开箱可用**：用户不需要自备 API Key；平台后端托管 Key，并做限流/配额。

---

## 2. 术语定义

- **clientId**：浏览器侧匿名标识（首次生成后保存在本地）。
- **familyId**：一个“家庭库存空间”的唯一 ID。
- **joinToken**：用于把新设备加入已有家庭的分享令牌（有编辑权限）。

---

## 3. 版本规划

## 3.1 v1（浏览器即用户）

### 目标
- 用户首次打开网页即可用（零注册、零登录）。
- 数据云端持久化，刷新不丢。

### 用户流程
1. 浏览器首次打开 → 生成 `clientId`。
2. 前端调用 `ensure` 接口，后端返回（或创建）一个 `familyId`。
3. 前端加载该家庭数据并渲染。
4. 用户增删改库存后，前端防抖上送后端。

### 特点
- 同一浏览器稳定使用体验好。
- 不自动跨设备（跨设备在 v2 通过共享加入实现）。

---

## 3.2 v2（家庭共享，全部可编辑）

### 目标
- 解决“同一个人多设备”与“家人共同管理同一家庭”。

### 用户流程
1. 旧设备在设置页点击“分享家庭” → 生成 `joinToken`（可做链接/二维码）。
2. 新设备打开链接（或扫码）→ 用 `joinToken` 调用 `join` 接口。
3. 新设备绑定到同一个 `familyId`，读取并编辑同一份库存。

### 权限策略
- 当前版本只有一种：**可编辑**（符合已确认需求）。
- 暂不做只读角色。

---

## 4. 架构方案（推荐）

## 4.1 前端
- 继续静态网页部署（Vercel 静态站点）。
- 新增“云同步层”（逻辑模块），负责：
  - 首次 `ensure`
  - 周期/事件触发的 `pull`
  - 防抖 `push`
  - 冲突提示
- 继续保留本地缓存（localStorage）以提升弱网体验。

## 4.2 后端
- 采用 Serverless API（Vercel Functions）：
  - 家庭管理与数据同步 API
  - 识别代理 API（OpenRouter）
- 数据库存储建议 Supabase/Postgres（可观测性和管理更友好）。

## 4.3 AI 服务
- OpenRouter Key 只在后端环境变量中保存（前端不可见）。
- 前端调用自家 `/api/recognize/*`，后端转发到 OpenRouter。

---

## 5. API 设计（最小可用）

> 下列为接口定义建议，后续可按实现语言细化为 OpenAPI。

## 5.1 家庭与绑定

### `POST /api/families/ensure`
- 入参：`{ clientId }`
- 行为：
  - 若 client 已绑定家庭：返回该家庭
  - 否则创建家庭并绑定
- 出参：`{ familyId, version, data }`

### `GET /api/families/:familyId`
- 入参：header 携带 `X-Client-Id`
- 出参：`{ familyId, version, data, updatedAt }`

### `PUT /api/families/:familyId`
- 入参：header `X-Client-Id`，body `{ baseVersion, data }`
- 行为：
  - 版本一致：写入成功并 `version+1`
  - 版本冲突：返回冲突错误
- 出参：`{ version, updatedAt }`

### `POST /api/families/:familyId/join-token`
- 入参：`X-Client-Id`
- 出参：`{ joinToken, joinUrl, expiresAt }`

### `POST /api/families/join`
- 入参：`{ clientId, joinToken }`
- 行为：把新 client 绑定到 token 对应家庭
- 出参：`{ familyId, version, data }`

## 5.2 AI 识别

### `POST /api/recognize/photo`
### `POST /api/recognize/order`
- 入参：图片（base64 或 multipart）
- 出参：识别 JSON（维持当前前端字段结构）
- 后端职责：
  - 调用 OpenRouter
  - 统一错误处理
  - 计费与限流

---

## 6. 数据模型（后端）

## 6.1 pantry_families
- `id` (uuid)
- `data_json` (jsonb)  // items、locations、可扩展元数据
- `version` (int)
- `updated_at` (timestamp)
- `created_at` (timestamp)

## 6.2 pantry_clients
- `client_id` (text, unique)
- `family_id` (uuid, fk)
- `created_at` (timestamp)
- `last_seen_at` (timestamp)

## 6.3 pantry_join_tokens
- `token` (text, unique)
- `family_id` (uuid, fk)
- `created_by_client_id` (text)
- `expires_at` (timestamp, 可选)
- `revoked_at` (timestamp, 可选)

## 6.4 pantry_recognition_usage（可选，但建议）
- `family_id`
- `client_id`
- `type` (`photo`/`order`)
- `created_at`
- `status` (`ok`/`error`)
- `error_code`（可空）

---

## 7. 同步策略（v1 先用快照）

## 7.1 写入
- 本地先改内存 + 渲染（保证交互顺滑）。
- 防抖（建议 500~1000ms）打一次 `PUT`。

## 7.2 冲突
- 后端基于 `baseVersion` 判断冲突。
- 冲突时前端提示：
  - “有其他设备更新，是否拉取最新并覆盖本地未提交改动”
  - 或自动拉取后再提示用户重试操作。

## 7.3 离线
- 离线时只写本地缓存，标记“待同步”。
- 恢复网络后自动重试同步。

---

## 8. 限流与成本控制（AI 开箱可用必需）

## 8.1 推荐限流维度
- `familyId`（主维度）
- `clientId`（次维度）
- `IP`（兜底风控）

## 8.2 推荐策略（初版）
- 每家庭：`photo` 每分钟 N 次、每日 M 次
- 每家庭：`order` 每分钟更低、每日更低（成本更高）
- 错误响应统一返回剩余额度提示

## 8.3 其他成本优化
- 保留现有前端图片压缩策略（降低 token 与带宽成本）
- 后端设置模型/超时上限，避免异常长请求

---

## 9. 安全与隐私

1. 前端不保存平台 OpenRouter Key。
2. joinToken 视为“家庭编辑凭证”，需使用高熵随机值。
3. joinToken 可设置有效期与撤销机制（v2 建议实现）。
4. 仅采集最低限度日志（避免存储敏感图片内容）。

---

## 10. 前端改造清单（不写实现代码，仅任务分解）

1. 统一入口文件（建议只保留 `pantry.html` 为主入口）。
2. 新增 `clientId` 生成/读取逻辑。
3. 新增云同步模块：
   - `ensureFamily()`
   - `pullFamily()`
   - `pushFamily()`
4. 设置页新增共享入口（生成链接/二维码）。
5. 识别请求改走后端 `/api/recognize/*`。
6. 保留 JSON 导入导出，作为备份/迁移工具。

## 11. 上线顺序建议（里程碑）

### Milestone A（1~2 天）
- 跑通 `ensure + get + put`，实现 v1 单浏览器云端持久化。

### Milestone B（0.5~1 天）
- 跑通 `joinToken + join`，实现 v2 多设备/多人共享编辑。

### Milestone C（0.5~1 天）
- 跑通后端 AI 识别代理与限流，前端去除用户自配 key 依赖。

### Milestone D（持续优化）

## 12. 当前已知事实与边界（避免后续误判）

1. 仓库内存在 `pantry.html` 与 `pantry-v4_1.html` 两个页面文件，需明确主入口（建议 `pantry.html`）。
2. 现阶段仍有部分功能在前端直连 OpenRouter（迁移到后端代理是后续目标）。
3. iPhone 端已做多轮交互与渲染优化，但仍需以真机回归作为上线标准。

---

## 13. 验收标准（Definition of Done）

v1 完成标准：
- 新浏览器打开即创建家庭并可保存数据
- 刷新后数据不丢失
- 无需登录

v2 完成标准：
- 旧设备可生成共享链接
- 新设备可加入同一家庭并看到同样库存
- 多端均可编辑并最终一致

AI 开箱可用标准：
- 用户不配置 key 也可识别
- 后端可限流，超限有明确提示
