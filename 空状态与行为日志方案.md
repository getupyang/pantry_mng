# 空状态首屏 & 行为日志方案

> 与当前产品能力对齐：`pantry.html` 已有 **FAB「＋」→ 录入页**（拍照识别 / 订单截图 / 手动填写）。本方案解决「首屏无物时空荡荡」与「补库存链路可观测」两件事。

---

## 一、首屏空状态（仪表盘 `items.length === 0`）

### 1.1 设计目标

- **一句话交代状态**：这个家清单里还没有任何物品。  
- **降低路径成本**：明确三种补入方式，并与现有入口一一对应（不发明新流程）。  
- **不遮挡常驻 FAB**：空状态是「内容区」的补充，右下角 **＋** 仍是全局主入口。

### 1.2 信息结构（建议）

| 区块 | 内容 |
|------|------|
| **主标题** | 「家里还没有物品」 |
| **副文案** | 「记下正在用的日用品，补货、临期会在首页提醒你。」 |
| **方式列表**（卡片或三行列表） | ① **拍照识别** — 拍包装正面，自动填品名、规格、保质期（推荐）<br>② **订单截图** — 电商订单图，可一次导入多件<br>③ **手动填写** — 在下面表单里逐项输入 |
| **主按钮** | 「去录入第一件」→ `nav('scr-add')`（与 FAB 同目标，给「不知道点哪」的用户一个超大靶心） |
| **次提示** | 「也可点右下角 ＋ 随时添加」 |

### 1.3 视觉与布局建议

- 仪表盘 **三张汇总卡可保留**（0 / 0 / 0），让用户理解「以后这里会亮」；或 **弱化为一条横条统计**，下方大面积留给空状态插画 + 文案（二选一，A/B 友好）。  
- 轻量插画：空架子 / 空购物袋线稿即可，避免抢主按钮。  
- **物品库 Tab**：若 `items.length===0`，同样展示简短空状态 +「去录入」，避免切 Tab 仍一片空白。

### 1.4 与「地点」的关系

- 首访若 **无地点**：可在空状态底部一行小字：「可在 **设置 → 地点管理** 添加存放位置；不填也可先记物品。」（与现有「未填写」地点选项一致，不阻断录入。）

## 二、行为日志（核心体验监控）

### 2.1 监控目的

| 关注点 | 说明 |
|--------|------|
| **识别链路健康** | 请求量、成功率、耗时、HTTP/业务错误码分布 |
| **识别质量** | 空结果、用户是否在 AI 预填后大量改字段（隐含「识别不准」） |
| **补库存完成率** | 从打开录入页 → 确认保存 的漏斗 |
| **同步可靠性** | 云同步失败次数（与「以为存了其实没上去」相关） |

### 2.2 设计原则

- **默认匿名**：与产品「无登录」一致；用已有 `clientId`、可选 `familyId`（UUID）关联，**不上传照片原图、不上传订单截图**。  
- **可聚合**：事件以 `event_name + 时间 + 维度` 为主，便于 Supabase / 日志仓做漏斗与告警。  
- **敏感字段**：品名/品牌等 **默认不入库**；若以后要分析「哪类品类识别差」，可用 **分类 slug**（如 `cat`）或 **哈希后的字段指纹**，需单独评审隐私政策。  
- **可靠投递**：先 **内存队列 + localStorage 环形缓冲**，空闲或页面前台时 **批量 POST**；失败重试、条数上限防爆本地。

### 2.3 事件命名规范

格式：`pantry.<域>.<动作>`，全小写，`.` 分层。

**域建议**：`intake`（补库存）、`recognize`（识别）、`sync`（同步）、`nav`（导航，可选）。

### 2.4 事件字典（v1 最小集）

以下列为 **建议首版必采**（覆盖你提到的失败、纠错）。

| event_name | 触发时机 | 建议 properties（JSON，均可选） |
|------------|----------|----------------------------------|
| `pantry.intake.screen_open` | 进入录入页 `scr-add` | `source`: `fab` \| `empty_state` \| `gallery_empty` \| `settings_import` |
| `pantry.intake.method_select` | 用户点「拍照 / 订单 / 仅手动」 | `method`: `photo` \| `order` \| `manual_only` |
| `pantry.recognize.start` | 开始调识别 API（含压缩后） | `kind`: `photo` \| `order`，`image_bytes_bucket`（如按 KB 分档：`<100k`） |
| `pantry.recognize.success` | 解析出非空可用结果 | `kind`，`latency_ms`，`field_count_filled`（非空字段数） |
| `pantry.recognize.fail` | 网络/5xx/解析抛错 | `kind`，`error_code`（应用内枚举），`http_status`，`latency_ms`，`message_hash`（错误文案 hash，便于聚类） |
| `pantry.recognize.empty_result` | 接口返回但业务判定「结果为空」 | `kind`，`latency_ms` |
| `pantry.intake.form_open` | 展示 AI 结果表单或纯手动表单 | `prefilled`: `ai` \| `none` |
| `pantry.intake.field_edit` | 用户修改某一字段（防抖合并） | `field`: `name` \| `brand` \| `size` \| `qty` \| `exp` \| `location`，`was_ai_prefilled`: bool |
| `pantry.intake.confirm_attempt` | 点击确认录入 | `prefilled` |
| `pantry.intake.confirm_success` | 本地保存成功 | `prefilled`，`item_id` 可用 uuid |
| `pantry.intake.confirm_fail` | 校验失败/保存异常 | `reason` 枚举 |
| `pantry.intake.correction_summary` | 确认时打点（**核心：一次保存一条摘要**） | `prefilled`，`edited_fields`: string[]，`ai_edit_count`（被改动的 AI 预填字段数） |

说明：**「识别错误用户手动改写」** 主要通过 `field_edit`（过程）+ `intake.correction_summary`（结果）两条一起看；报表可看「`prefilled=ai` 且 `ai_edit_count>=2` 的占比」。

### 2.5 同步与其它（v1 建议带上）

| event_name | 触发时机 | properties |
|------------|----------|------------|
| `pantry.sync.push_success` | PUT 家庭数据成功 | `latency_ms`，`item_count` |
| `pantry.sync.push_fail` | PUT 失败 | `http_status`，`reason` |
| `pantry.sync.conflict_409` | 版本冲突 | — |
| `pantry.import.json_success` / `json_fail` | 设置里导入 JSON | `item_count` / `reason` |

### 2.6 服务端与存储（实现时参考）

1. **表（Supabase）**：例如 `pantry_behavior_events`  
   - `id` uuid，`created_at` timestamptz  
   - `client_id` text，`family_id` uuid nullable  
   - `event_name` text，`props` jsonb  
   - `app_version` text（build 或 `pantry.html` 注释版本号）  
2. **API**：`POST /api/behavior/batch`，body `{ events: [{ name, props, ts }] }`，服务端补 `received_at`，**限流**（按 `client_id`）。  
3. **看板**：Metabase / Supabase SQL + 定时查询；告警：`recognize.fail` 率、`empty_result` 率、确认漏斗骤降。

### 2.7 与现有 `pantry_recognition_usage` 的关系

- `pantry_recognition_usage`：偏 **计费/配额**（按日、按家庭或客户端计数）。  
- `pantry_behavior_events`：偏 **产品与体验**（漏斗、字段纠错、错误类型）。  
- 两者可同时写：识别成功时 usage +1，同时发 `recognize.success` 事件。

## 三、落地顺序建议

1. **空状态 UI**（仅 `items.length===0` 分支，不改其它列表逻辑）。  
2. **前端 `track()` 封装** + 队列；先 `debug` 模式可打 console。  
3. **批量上报 API + 表**；上线后先看 7 日 `recognize.*` 与 `correction_summary` 分布。
